# 归档历史 (Archive History)

## 目录说明
此目录用于存放已完成、已合并、已验证的历史变更包。

## 目录状态
- **状态**: 只读 (Read-Only)
- **内容**: 历史变更的完整记录
- **用途**: 审计追踪、知识传承、问题排查

## 归档目录结构
```
archive/
└── <日期>-<change-id>/     # 归档目录，格式: YYYY-MM-DD-<change-id>
    ├── proposal.md         # 原始提案文档
    ├── design.md           # 原始设计文档
    ├── tasks.md            # 原始任务清单
    └── specs/              # 原始规范增量
        ├── <module-a>/
        │   └── spec.md     # 模块 A 规范修改历史
        └── <module-b>/
            └── spec.md      # 模块 B 规范修改历史
```

## 命名规范
- **归档目录**: `<日期>-<change-id>`
- **日期格式**: YYYY-MM-DD
- **示例**: `2026-03-23-add-user-login`, `2026-03-24-fix-payment-bug`

## 归档内容
每个归档目录包含变更的完整历史记录：

### 文档完整性
- **四文档完整**: proposal.md, design.md, tasks.md, specs/
- **代码快照**: 变更完成时的代码状态（通过版本控制追踪）
- **测试结果**: 变更验证通过时的测试报告

### 元数据信息
- **归档日期**: 变更完成的日期
- **变更作者**: 负责此变更的开发者
- **审查记录**: 代码审查和批准记录
- **验证结果**: 测试覆盖率、性能指标等

## 使用场景

### 审计追踪
- 追踪特定功能的演进历史
- 分析代码变更对系统的影响
- 排查历史问题的根本原因

### 知识传承
- 新成员了解系统演进过程
- 学习过往的设计决策和实现方案
- 避免重复犯同样的错误

### 问题排查
- 当发现规范漂移时，对比历史版本
- 分析性能回归问题的引入时间点
- 排查安全漏洞的历史根源

## 维护规则

### 归档时机
- 变更通过所有验证测试后立即归档
- 归档前必须确保四文档完整且准确
- 归档后不得修改归档内容

### 访问权限
- **读取权限**: 所有开发者可读取历史归档
- **修改权限**: 禁止任何人对归档内容进行修改
- **删除权限**: 仅系统管理员可删除归档（需记录原因）

### 存储管理
- 定期清理过期的归档（如超过2年的归档）
- 重要变更的归档应永久保留
- 归档目录大小应控制在合理范围内

## 最佳实践

### 归档质量
- 确保归档内容完整可读
- 包含足够的技术上下文信息
- 记录重要的设计决策和权衡

### 检索优化
- 使用有意义的变更ID便于搜索
- 在归档目录中添加 README 说明变更概要
- 建立索引文件记录所有归档变更

> **重要**: 归档是项目的历史财富，应妥善保管和维护。每个归档都代表了项目演进的一个重要里程碑。
