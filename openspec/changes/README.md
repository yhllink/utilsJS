# 变更工作区 (Change Workspace)

## 目录说明
此目录用于存放所有进行中的变更工作。每个子文件夹代表一个独立的变更单元。

## 目录状态
- **状态**: 读写 (Read-Write)
- **使用范围**: 仅限当前活跃开发
- **生命周期**: 变更完成后必须归档

## 变更目录结构
```
changes/
└── <change-id>/          # 变更单元目录，命名格式: <动作>-<对象>
    ├── proposal.md       # [必需] 变更提案文档
    ├── design.md         # [必需] 技术设计文档
    ├── tasks.md          # [必需] 任务执行清单
    └── specs/            # [必需] 规范增量目录
        ├── <module-a>/
        │   └── spec.md   # 对模块 A 的规范修改草案
        └── <module-b>/
            └── spec.md   # 对模块 B 的规范修改草案
```

## 命名规范
- **Change ID**: `<动作>-<对象>` (kebab-case)
- **示例**: `add-user-login`, `fix-payment-bug`, `optimize-sort-performance`

## 四文档要求
每个变更必须包含以下四份核心文档：

### 1. proposal.md (提案文档)
- 定义"为什么做"和"做什么"
- 必须包含背景、目标、范围、验收标准
- 验收标准必须使用 Gherkin 格式

### 2. design.md (设计文档)
- 定义"怎么做"
- 必须包含架构影响、数据模型、技术选型
- 必须包含架构图或数据流图

### 3. tasks.md (任务清单)
- 定义"步骤顺序"
- 必须使用复选框格式，每个任务有唯一 ID
- AI 代理必须严格按此顺序执行

### 4. specs/ (规范增量)
- 定义"系统应如何表现"
- 必须标注与原有规范的差异点
- 是生成代码和测试用例的直接依据

## 工作流程
1. **创建变更目录**: 使用标准命名格式
2. **编写四文档**: 确保内容完整准确
3. **执行任务**: 严格按 tasks.md 顺序执行
4. **验证变更**: 运行测试确保功能正确
5. **归档完成**: 将变更合并到稳定规范

## 禁止行为
- ❌ 直接修改 openspec/specs/ 中的文件
- ❌ 跳过四文档直接编写代码
- ❌ 在未完成验证的情况下归档
- ❌ 忽略任务执行顺序

> **重要**: 所有开发工作必须限制在此目录内，严禁直接修改稳定规范。
